# 01. 问题陈述：为什么需要 Memory

---

## 1.1 Claude Code 的"失忆"问题

Claude Code 存在三个致命的记忆缺陷：

```
1. /compact 之后 → 之前的决策、发现、上下文全部丢失
2. 跨 session → 新会话对旧会话一无所知
3. 长会话 → 上下文窗口塞满，早期关键信息被挤出去
```

对于需要"持续学习用户思维、成为用户延伸"的数字分身来说，没有可靠的长期记忆等于没有灵魂。

---

## 1.2 "学习"能力的四层拆解

```
┌──────────────────────────────────────────────────┐
│ 1. 数据采集层  ← 从各渠道持续读取内容              │
│    Twitter / WeChat / RSS / 网页 / 文档 / ...     │
├──────────────────────────────────────────────────┤
│ 2. 知识处理层  ← 理解、提取、摘要、分类            │
│    NLP / LLM 提取关键洞察                         │
├──────────────────────────────────────────────────┤
│ 3. 知识存储层  ← 持久化 + 关联 + 检索              │
│    图谱 / 向量 / 全文搜索                          │
├──────────────────────────────────────────────────┤
│ 4. 知识应用层  ← Agent 按需调用、推理、输出         │
│    Claude 基于积累的知识回答/决策                   │
└──────────────────────────────────────────────────┘
```

Mem0 主要覆盖第 2-3 层（知识处理 + 存储），MemCP 主要覆盖第 3-4 层（存储 + 应用）。两者组合可覆盖全链路。

---

## 1.3 为什么不直接用 Mem0

| 维度 | Mem0 | MemCP + Claude Code CLI |
|------|------|------------------------|
| **依赖** | Python 3.12+、PostgreSQL、LLM API Key（OpenAI/OpenRouter） | Python + SQLite（零外部依赖） |
| **部署** | Docker + 数据库 + 配置 API Key | `pip install memcp`，开箱即用 |
| **图谱能力** | Neo4j 单一关系边 | MAGMA 4-图（语义/时间/因果/实体） |
| **LLM 推理** | 内置 GPT-4o-mini（需付费） | 复用 Claude Code CLI（已有） |
| **可控性** | 黑盒 Pipeline | System Prompt 可随时调整 |
| **成本** | Mem0 + OpenAI 双重付费 | MemCP 零成本 + Claude 已有开销 |

**结论**：与其引入 Mem0 的完整技术栈，不如用 MemCP 做存储层，用 Claude Code CLI 的 LLM 能力补齐 Mem0 独有的智能处理层。

---

## 1.4 传统 RAG/Memory 方案的结构性瓶颈

1. 是"LLM 的附属品"而非独立实体——记忆生命周期绑定 agent 会话
2. 写入是被动的（对话结束后提取摘要），reactive 模式丢失结构信息
3. 检索是扁平的（vector similarity），无法表达时间/因果/矛盾
4. 没有遗忘机制（全记或 TTL 一刀切），无智能衰减
5. 跨 agent 共享是后加的，不是原生的
