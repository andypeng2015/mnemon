# 14. 去中心化架构适配性分析

---

## 14.1 各层适配度

| 层级 | 去中心化适配性 | 原因 |
|------|-------------|------|
| 写入层 | ✅ 天然适合 | "宽进"不需要全局共识，各节点独立写入本地 log，eventual consistency |
| 整理层 | ⚠️ 存在张力 | 跨节点关联需要全局视野，关联质量和可见范围正相关 |
| 检索层 | ✅ 有优势 | 查询天然局部化，先搜本地再扩展远端，延迟和隐私双优 |

---

## 14.2 可行模式

- **联邦式：** 各节点维护本地图谱，定期向聚合层同步摘要（不是原始数据），聚合层做跨节点关联
- **Gossip 式：** 节点间 P2P 交换实体索引，逐步发现跨节点共同实体，收敛慢但无中心
- **Git 模式（最 pragmatic）：** 本地优先，云端可选同步。本地是完整的，push/pull 是可选的

---

## 14.3 建议路径

去中心化对现阶段是过度设计。优先级：

```
v0.1  单机 daemon，SQLite，跑通闭环
v0.5  支持多 agent 读写同一图谱（多维护者）
v1.0  本地优先 + 可选云同步（类 Git 模型）
v2.0  多节点联邦式整合（如果确实有需求）
```

**一个节点的记忆如果都不好用，十个节点只是十倍的垃圾。**
