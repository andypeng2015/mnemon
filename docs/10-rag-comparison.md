# 10. 与 RAG 方案的定位对比

---

## 10.1 核心问题：这是 RAG 吗？

**部分是，但整体不是。**

把我们的架构拆成读路径和写路径来看：

```
┌─────────────────────────────────────────────────────────────┐
│                     我们的完整架构                            │
│                                                             │
│  写路径（记忆实例）              读路径（主实例）              │
│  ┌───────────────────┐         ┌───────────────────┐       │
│  │ 事实提取            │         │ memcp_recall       │       │
│  │ 智能去重            │         │ memcp_related      │       │
│  │ 冲突解决            │         │ Qdrant search      │       │
│  │ 生命周期管理        │         │ → 注入上下文        │       │
│  │                   │         │ → 生成回复          │       │
│  │ 这不是 RAG         │         │ 这是 RAG            │       │
│  │ 这是知识工程        │         │ （高级变体）         │       │
│  └───────────────────┘         └───────────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

读路径确实是 RAG——查询 → 检索 → 注入上下文 → 生成回复。但写路径完全超出了 RAG 的范畴。**RAG 假设知识库是静态的或人工维护的**，我们的知识库是 LLM 自主维护的。

更准确的定位：**我们的架构是一个认知记忆系统（Cognitive Memory Architecture），其中包含 RAG 作为读取组件。**

---

## 10.2 架构模式对比

| 方案 | 读路径 | 写路径 | 知识维护 |
|------|--------|--------|---------|
| **Naive RAG** | 向量搜索 → 拼接 → 生成 | 人工索引文档 | 无 |
| **Advanced RAG** | 查询改写 + 多策略检索 + 重排序 | 人工索引文档 | 无 |
| **Microsoft Graph RAG** | 图遍历 + 社区摘要 | LLM 批量建图（一次性） | 需重建整个图 |
| **Agentic RAG** | Agent 自主决策检索策略 | 人工索引文档 | 无 |
| **Self-RAG** | 检索 + 自我反思 + 质量评估 | 人工索引文档 | 无 |
| **RAPTOR** | 递归树状摘要检索 | 批量构建摘要树 | 需重建 |
| **我们的架构** | Agent 导航图谱 + 向量搜索 | **LLM 持续提取/去重/冲突解决** | **自动生命周期** |

关键发现：**所有现有 RAG 方案的写路径都是被动的**（人工或一次性批处理）。我们是唯一拥有持续活跃写路径的。

## 10.4 与 Microsoft Graph RAG 深度对比

Graph RAG 是与我们最相似的方案，值得单独对比：

```
Microsoft Graph RAG                    我们的架构
┌────────────────────┐                ┌────────────────────┐
│ 批量建图阶段（离线）  │                │ 持续记忆（在线）     │
│                    │                │                    │
│ 文档 → LLM 提取    │                │ 对话 → LLM 提取    │
│ 实体+关系           │                │ 事实+关系           │
│      ↓             │                │      ↓             │
│ 构建 Entity Graph  │                │ 更新 MAGMA 4-图    │
│      ↓             │                │      ↓             │
│ Leiden 社区检测     │                │ 自动连边（4 种类型） │
│      ↓             │                │      ↓             │
│ 社区摘要生成        │                │ 去重 + 冲突解决     │
│                    │                │                    │
│ 查询阶段（在线）     │                │ 查询阶段（在线）     │
│ Global: 社区摘要    │                │ 图遍历 + 向量搜索   │
│ Local: 实体邻居     │                │ memcp_related      │
│                    │                │ + memcp_recall     │
└────────────────────┘                └────────────────────┘
```

| 维度 | Microsoft Graph RAG | 我们的架构 |
|------|--------------------|-----------|
| **图类型** | 单一实体-关系图 | MAGMA 4-图（语义/时间/因果/实体） |
| **建图方式** | 批量一次性 | 持续增量 |
| **更新成本** | 需重建整个图 | 增量 remember/forget |
| **检索策略** | Global（社区摘要）+ Local（邻居） | 图遍历（4 种边）+ 向量 + 5 级降级 |
| **知识维护** | 无（静态图） | 自动去重、冲突解决、retention lifecycle |
| **适用场景** | 大规模文档库的全局问答 | 个人知识积累与持续学习 |
| **依赖** | LLM API + Neo4j/等 | 零外部依赖（MemCP 纯本地） |

**Graph RAG 更擅长**：大规模文档库的"全局性"问题（"这个领域的主要主题是什么？"），通过社区摘要可以回答跨越整个语料库的问题。

**我们更擅长**：持续演化的个人知识库、因果推理、时间线追溯、矛盾信息处理。

## 10.5 成本模型对比

| 方案 | 构建成本 | 查询成本 | 维护成本 | 总 TCO 特征 |
|------|---------|---------|---------|------------|
| **Naive RAG** | 低（embedding 一次） | 低（一次检索+生成） | 手动重索引 | 低但人工成本隐性高 |
| **Advanced RAG** | 低 | 中（多步检索） | 手动 | 中 |
| **Graph RAG** | **极高**（LLM 处理全量文档） | 中 | 需重建 = 再次极高 | 前高后低，更新痛苦 |
| **Agentic RAG** | 低 | 高（多轮 Agent 调用） | 手动 | 查询成本高 |
| **我们的架构** | 零 | 低（主实例零额外开销） | **自动，用 Haiku** | 持续低成本 |

Cowork 架构的成本优势在这里非常明显：
- Graph RAG 需要用 GPT-4 级模型处理全量文档来建图，成本极高
- 我们的记忆实例用 Haiku 做提取/去重（Opus 成本的 1/60），且是增量的
- 主实例查询时只做 recall/related，不触发记忆管理流程

## 10.6 在 RAG 演进谱系中的位置

```
RAG 演进谱系：

Naive RAG → Advanced RAG → Modular RAG → Agentic RAG → Graph RAG
                                                           │
                                                    ┌──────┘
                                                    ↓
                                            我们在这里
                                                    │
                                            ┌───────┴───────┐
                                            │ 超越 RAG 的部分 │
                                            │               │
                                            │ • 持续写路径   │
                                            │ • 去重/冲突    │
                                            │ • 生命周期     │
                                            │ • Cowork 分离  │
                                            └───────────────┘
```

**一句话定位**：读路径是 **Agentic Graph RAG**（Agent 自主导航知识图谱检索），写路径是 **Automated Knowledge Engineering**（LLM 驱动的持续知识管理），两者通过 **Cowork 架构异步解耦**。

## 10.7 与成熟 RAG 方案相比的不足

| 不足 | 说明 | 可借鉴的 RAG 方案 | 优先级 |
|------|------|-----------------|--------|
| **无检索质量自评** | 不知道检索到的记忆是否真正相关 | Self-RAG 的 reflection tokens | Phase 3 |
| **无全局摘要能力** | 不能回答"我所有知识的全局概览是什么" | Graph RAG 的社区摘要 | Phase 3 |
| **无查询自适应** | 不会根据查询复杂度动态选择检索策略 | Adaptive RAG | Phase 3 |
| **图谱规模未经验证** | MAGMA 4-图在万级以上的性能未知 | Graph RAG 的 Leiden 社区分层 | Phase 2 |
| **提取质量无反馈闭环** | 记忆实例提取了错误事实，没有机制自动发现和纠正 | Self-RAG 的 critique model | Phase 3 |

这些不足是 Phase 3（闭环优化）需要重点补齐的方向。特别是 Self-RAG 的检索质量自评机制——让主实例在使用记忆时评估检索质量并反馈给记忆实例，可以形成记忆质量的闭环改进。
